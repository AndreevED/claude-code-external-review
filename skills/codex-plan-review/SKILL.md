# Codex Plan Review

Независимое ревью планов реализации через Codex: план отправляется на проверку, Codex исследует кодовую базу и возвращает структурированные замечания, ты исправляешь план, и цикл повторяется до одобрения или лимита итераций.

## Формат вызова

```
/codex-plan-review путь/к/плану.md [максимум N итераций]
```

Если путь не указан — спроси у пользователя.
Если максимум итераций не указан — по умолчанию 5.

## Алгоритм

### 1. Подготовка

1. Прочитай файл плана через Read
2. Создай папку `.codex-plan-review/` в корне проекта — в неё будут писаться промпты, лог запусков и лог ревью
3. Инициализируй лог ревью: файл `.codex-plan-review/<имя-плана>-codex-review-log.md`
4. Собери контекст правил проекта (подробности в разделе «Сборка контекста правил»)

### 2. Сборка контекста правил

Codex не читает `.claude/rules/` и `CLAUDE.md` автоматически — контекст нужно передать явно в промпте.

1. Прочитай все файлы из `~/.claude/rules/*.md` (глобальные правила пользователя)
2. Прочитай `~/.claude/CLAUDE.md` (глобальный CLAUDE.md пользователя, если существует)
3. Найди и прочитай все файлы из `.claude/rules/*.md` в корне текущего проекта (проектные правила, если каталог существует)
4. Найди и прочитай `CLAUDE.md` в корне текущего проекта (если существует). Также проверь наличие `CLAUDE.md` в текущем рабочем каталоге, если он отличается от корня проекта
5. Прочитай файл `references/review-system-prompt.md` относительно этого скилла — используй абсолютный путь
6. Объедини правила (без system prompt) в переменную `RULES_CONTEXT` в формате:

```
--- Файл: имя_файла_правил.md ---
содержимое файла

--- Файл: ~/.claude/CLAUDE.md ---
содержимое глобального CLAUDE.md

--- Файл: CLAUDE.md ---
содержимое проектного CLAUDE.md
```

Если какие-то файлы не найдены — просто пропусти их, это не ошибка.

### 3. Цикл ревью

Повторяй до тех пор, пока Codex не вернёт `"status": "approved"` или не исчерпан лимит итераций:

**a) Собери промпт для Codex:**

Промпт строится из трёх блоков — system instructions, rules, задание:

```
<system_instructions>
СОДЕРЖИМОЕ_REVIEW_SYSTEM_PROMPT_MD
</system_instructions>

<project_rules>
RULES_CONTEXT
</project_rules>

ПРОМПТ_ИТЕРАЦИИ
```

**Промпт первой итерации:**
```
Проведи ревью плана реализации. Файл: АБСОЛЮТНЫЙ_ПУТЬ_К_ПЛАНУ
```

**Промпт последующих итераций:**
```
Проведи ревью плана реализации. Файл: АБСОЛЮТНЫЙ_ПУТЬ_К_ПЛАНУ

Следующие замечания уже рассмотрены — не повторяй их:
<accepted_issues>
НАКОПЛЕННЫЙ_СПИСОК_ПРИНЯТЫХ_ЗАМЕЧАНИЙ
</accepted_issues>

<rejected_issues>
НАКОПЛЕННЫЙ_СПИСОК_ОТКЛОНЁННЫХ_ЗАМЕЧАНИЙ
</rejected_issues>

Сосредоточься на новых проблемах, которых ещё нет в этих списках.
```

`НАКОПЛЕННЫЙ_СПИСОК_ПРИНЯТЫХ_ЗАМЕЧАНИЙ` — компактный список всех CONFIRMED замечаний за все итерации, по одной строке: `[Секция]: суть в 5-7 словах`. Например:
```
[Шаг 2]: РежимЗавершения в структуре результата
[Шаг 3]: ЭтоОбработкаPostMessage в сигнатуре функции
[Шаг 4]: убран ДокументЭДО, заменён НетДокументаЭДО+Данные1С
```

`НАКОПЛЕННЫЙ_СПИСОК_ОТКЛОНЁННЫХ_ЗАМЕЧАНИЙ` — компактный список всех FALSE POSITIVE замечаний, по одной строке: `[Секция]: суть — причина отклонения в 3-5 словах`. Например:
```
[Шаг 3]: МЧД_В_1С внутри функции — в коде уже снаружи
```

Если отклонённых замечаний нет — блок `<rejected_issues>` можно опустить.

**b) Запусти codex:**

Запиши полный промпт в файл `.codex-plan-review/prompt-iter<N>.txt` через Write, затем передай через stdin:

```bash
RESULT=$(codex exec \
  -s read-only \
  --dangerously-bypass-approvals-and-sandbox \
  - < ".codex-plan-review/prompt-iter<N>.txt" \
  2>".codex-plan-review/run.log") \
  || (echo "codex failed:"; cat ".codex-plan-review/run.log")
echo "$RESULT"
```

- Передача через файл+stdin надёжнее прямого аргумента: промпт содержит теги, кавычки и другие спецсимволы
- stderr (лог сессии) идёт в `.codex-plan-review/run.log` — не засоряет контекст
- Финальный ответ агента codex пишет в **stdout** — читай из `$RESULT`
- Установи таймаут Bash-вызова в 600000 мс (10 минут)

**c) Распарси ответ:**

JSON читай из переменной `$RESULT` (stdout bash). Если обёрнут в markdown-блок (` ```json ... ``` `), извлеки содержимое между маркерами.

Структура ответа Codex:
```json
{
  "status": "approved" | "has_issues",
  "issues": [
    {
      "severity": "critical" | "important",
      "confidence": 75,
      "section": "Секция плана",
      "problem": "Описание проблемы",
      "suggestion": "Предложение по исправлению",
      "evidence": "Подтверждение из кода или логическое обоснование"
    }
  ],
  "summary": "Общая оценка"
}
```

**d) Если `status == "approved"` или `issues` пуст — выйди из цикла.**

**e) Stalemate detection:** если замечания текущей итерации по сути повторяют предыдущие (те же секции, те же проблемы) — прерви цикл и сообщи пользователю о расхождении мнений.

**f) Если есть новые замечания:**
1. Верифицируй каждое замечание — прочитай упомянутый код, проверь факты. Классифицируй: CONFIRMED / FALSE POSITIVE
2. Исправь план по подтверждённым замечаниям (используй Edit для файла плана)
3. Добавь CONFIRMED замечания в накопленный список `accepted_issues` (`[Секция]: суть`)
4. Добавь FALSE POSITIVE замечания в накопленный список `rejected_issues` (`[Секция]: суть — причина`)
5. Допиши итерацию в лог-файл
6. Перейди к следующей итерации

Между итерациями не показывай пользователю промежуточные результаты — работай молча до завершения.

### 4. Итоговый отчёт

После завершения цикла выведи пользователю:

```
## Результат ревью (Codex)

- Итераций: N из M (максимум)
- Найдено проблем: X (критических: Y, важных: Z)
- Статус: план одобрен / лимит итераций достигнут / stalemate
- Лог ревью: путь/к/лог-файлу.md
```

Если лимит итераций достигнут:
- Все замечания последней итерации были CONFIRMED и исправлены → завершить, написать "лимит достигнут, все найденные проблемы исправлены"
- Есть неисправленные замечания (FALSE POSITIVE или stalemate) → показать что осталось и спросить пользователя что делать дальше

## Формат лога

Файл `.codex-plan-review/<имя-плана>-codex-review-log.md`:

```markdown
# Лог ревью (Codex): <имя файла плана>
Дата: <дата>
Максимум итераций: N

## Итерация 1

### Замечания Codex
- [critical, 95] Секция: ... — Проблема → Предложение
- [important, 80] Секция: ... — Проблема → Предложение

### Верификация
- CONFIRMED: описание → исправлено
- FALSE POSITIVE: описание → причина отклонения

### Принято / Отклонено
- ПРИНЯТО: [Секция]: суть
- ОТКЛОНЕНО: [Секция]: суть — причина

## Итерация 2
...

## Итог
- Всего итераций: N
- Всего найдено проблем: M (подтверждено: K, отклонено: L)
- Статус: одобрен / лимит достигнут / stalemate
```

## Обработка ошибок

- **`$RESULT` пустой или не валидный JSON**: повтори вызов один раз. Если снова ошибка — покажи содержимое `.codex-plan-review/run.log` пользователю
- **Файл плана не найден**: сообщи пользователю, предложи указать путь
- **codex не найден в PATH**: сообщи пользователю, что команда `codex` не найдена

## Важные моменты

- Codex НЕ получает rules и CLAUDE.md автоматически — ты передаёшь их в промпте через блок `<project_rules>`
- При исправлении плана используй свои знания и контекст проекта, не копируй слепо предложения Codex — верифицируй каждое замечание
- Не показывай промежуточные результаты — пользователь увидит итоговый отчёт и лог-файл
